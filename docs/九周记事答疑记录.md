# 毕设九周记事答疑记录（详细版）

## 第1周
**每周工作记事**  
本周完成了项目基础环境的搭建与可用性验证。首先在本地创建并配置 `llama3` 环境，安装项目依赖，检查 `torch + CUDA` 是否可用；随后启动推理服务脚本，验证模型 API 端口连通性与基础请求响应。为后续排查问题，本周同步建立了“命令-日志-结果文件”三点对应的记录习惯，保证每次运行都能追溯到具体输入与输出。  

**问题简述**  
环境初始化阶段出现 `bitsandbytes` 提示信息与系统实际情况不完全一致的问题，日志中出现 Linux 风格路径与 Windows 运行状态混杂，容易误判为 GPU 不可用。  

**教师回答**  
建议以“服务能否稳定启动并返回有效结果”作为首要判据，而不是仅依赖单条依赖库提示。要求保留完整日志，并在后续形成可复用的环境检查清单，降低重复排错成本。  

---

## 第2周
**每周工作记事**  
进入第1阶段（实体对筛选）调试，完成 `prompt -> run -> check -> label` 的完整链路验证。通过 0-5 小样本跑通阶段脚本，确认输入文档、实体列表、模型输出与最终标签文件之间的关联关系。同时梳理了阶段产物命名规则，为后续多阶段串联做准备。  

**问题简述**  
运行中出现两类问题：一是模型输出中含特殊结束标记，导致解析脚本报 `SyntaxError`；二是不同目录启动脚本时相对路径不一致，出现“同一命令在不同位置结果不同”的情况。  

**教师回答**  
先固定运行入口目录，统一相对路径假设；对模型回复增加格式清洗与容错逻辑，避免单条异常阻断全流程。优先保证小样本端到端稳定，再扩到更大范围。  

---

## 第3周
**每周工作记事**  
本周完成第2阶段（实体信息）与第3阶段（关系摘要）的串联调试，初步形成“实体对标签驱动后续语义生成”的工作流。通过逐步验证提示文件、运行分片与汇总结果，确保每个阶段都能稳定落盘，并可被下游脚本读取。  

**问题简述**  
出现 `ModuleNotFoundError`、目标输出目录不存在、部分脚本输入文件命名不一致等问题，导致阶段中断。  

**教师回答**  
建议统一模块引用方式与工程根目录约定，所有需要输出的路径在脚本中自动创建；并形成“先检查输入文件存在，再执行主逻辑”的防御式流程。  

---

## 第4周
**每周工作记事**  
推进第4阶段检索流程，先完成训练侧小范围缓存构建（`train_annotated`），再连接 `dev` 侧检索。重点验证 embedding 文件与关系摘要文件是否一一对应，并检查检索输出的候选关系格式是否满足下游要求。  

**问题简述**  
检索阶段存在训练缓存缺失、字段名不统一（如 `label_rel` 读取异常）、以及缓存文件选择错误（多版本并存时读取了旧文件）的问题。  

**教师回答**  
要求先明确缓存优先级与版本选择规则，保证“同一范围同一版本”被稳定读取；同时建议补充缓存一致性检查，减少后续链路中的隐性数据错配。  

---

## 第5周
**每周工作记事**  
本周完成第5阶段（多选精筛）和第6阶段（事实判定）的小样本闭环，首次实现从原始文档到最终三元组文件的端到端输出。开始引入脚本化运行思路，尝试将多阶段命令封装为统一流水线执行。  

**问题简述**  
长链路运行过程中出现参数类型转换错误、目录缺失引发写入失败、步骤失败后手动重跑成本高等问题，影响实验连续性。  

**教师回答**  
明确提出先做工程稳定性：增加分阶段保存、失败中断定位与可恢复机制；将“可持续运行”作为阶段目标，再进行指标优化。  

---

## 第6周
**每周工作记事**  
重点建设自动化流水线与容错能力。完成 `dev` 与 `train_annotated` 两条脚本链路的断点续跑机制，加入检查点文件保存、分片自动汇总、日志分流（stdout/stderr）等功能。经过多轮验证，流程在中断后可从已完成阶段继续，显著降低重复计算。  

**问题简述**  
全量运行时暴露边界问题：编码异常（GBK/UTF-8混用）、检查点文件偶发损坏、恢复状态与实际产物不一致。  

**教师回答**  
建议采用“原子写入 + 备份检查点 + 回退读取”的方式处理状态文件；并将阶段完成标志与产物完整性联合校验，确保恢复逻辑可靠。  

---

## 第7周
**每周工作记事**  
在原六阶段流程上接入创新模块：证据感知关系重排（EAR-Rerank），完成 baseline 与 rerank 双路径切换配置。跑通 0-5 对照实验并记录过程指标（阶段候选数量、最终预测数量），形成初步增益分析框架。  

**问题简述**  
小样本结果波动较大，部分实验中仅看到候选分布变化而最终 F1 提升不明显，难以直接形成稳定结论。  

**教师回答**  
小样本主要用于“功能正确性与链路验证”，不用于最终性能结论。要求保留过程指标，后续在更大样本和全量数据上做正式对比。  

---

## 第8周
**每周工作记事**  
完成 `dev` 全量评测与误差统计，输出 TP/FP/FN、Precision/Recall/F1，并进行阶段留存分析，定位召回损失主要集中在前段候选筛选。同步对照论文报告结果，分析实现结构与实验配置的差异来源。  

**问题简述**  
与目标论文相比仍有明显差距，存在“是否实现错误”与“是否结构不一致”两类疑问。  

**教师回答**  
先确认工程实现正确，再讨论指标差距。当前差异的核心在于尚未实现论文同构的 RCP（二分类预分类器）+RC 主结构，后续应优先补齐该模块并按官方口径评测。  

---

## 第9周
**每周工作记事**  
本周完成中期材料汇总：系统架构图、数据流转表、实验结果表、九周记事答疑记录与阶段性开发记录。对已有产物进行可追溯整理，做到“每个结论可对应脚本、日志与结果文件”。同时制定后续计划：补齐论文同构模块、完善评测对齐、推进更大规模实验。  

**问题简述**  
中期汇报不仅要体现“能跑通”，还要体现“方法价值与后续可提升空间”，需要将调试细节提炼成规范、可展示的阶段成果。  

**教师回答**  
建议按“已完成能力—当前短板—下一步路线”组织汇报主线；图表以可复核、可追溯为原则，避免仅给出结论而无过程证据。  

---

## 阶段性总体结论
目前项目已实现：  
1. 六阶段主流程可运行且可复现；  
2. 具备断点续跑、分阶段落盘、日志追踪能力；  
3. 完成 baseline/rerank 的可对照评测框架。  

下一阶段重点为：补齐论文同构的 RCP 二分类预分类器、完成官方口径对齐评测、在更大样本上形成稳定且可解释的性能结论，为论文实验章节提供完整支撑。
